## 2. Invariants in plain English

What MUST hold:

1. **Identity of the protocol.** Exactly one NFT under `one_shot_mint` exists, lives at `reference_holder` forever, and carries an inline `ReferenceDatum` that no validator is allowed to ever change.
2. **Ownership.** A mix-box with registers `(a, b)` can be spent via the Owner branch only by a party that knows `x` such that `b = [x]·a`.
3. **Re-randomization preserves owners.** A Mix tx maps each input `i` to one output position with `(a', b') = ([y_i]a_i, [y_i]b_i)`. The sigma-OR proves "this input got mapped to one of the N outputs", without revealing which.
4. **Value preservation in mix and fee paths.** Every mix output carries exactly `denom_lovelace` ada and no native assets. Every fee output preserves the unit datum and is ada-only. PayMixFee: `fee_in − fee_out == tx.fee` and `tx.fee ≤ max_fee_per_mix_lovelace`. Replenish: `fee_out > fee_in`.
5. **Fee shard can only be spent by a real Mix tx (or a Replenish).** PayMixFee gates on the `mix_logic` withdraw redeemer being `Mix { .. }`. An Owner-branch withdraw cannot consume a fee shard regardless of its input mix.
6. **Fiat-Shamir context binding.** Owner proofs commit to `serialise_data(tx.outputs) || mix_script_hash`. Mix proofs commit to the N output datums + N output values (= `denom_lovelace` ada-only every time) + `mix_script_hash`. Output substitution invalidates every proof.
7. **Stake-credential lifecycle.** The `mix_logic` stake credential, once registered, can never be unregistered/delegated/voted/etc., `mix_logic.publish` accepts `RegisterCredential` only.
8. **Hyperstructure permanence.** `reference_holder.spend = False`, NFT is one-shot, all parameters are pulled from the inline datum at validation time. No admin key, no upgrade authority.

What is intentionally terminal-True under hyperstructure (Rule 2):

- A `mix_box` UTxO whose datum doesn't decode as well-formed inline `MixDatum` (incl. `DatumHash(_)`, `NoDatum`, malformed inline shape) is freely spendable.
- A `fee_contract` UTxO whose datum is anything other than `Constr 0 []` is freely spendable.

## 3. Adversarial findings

Each finding states: severity, confidence, file, violated invariant (if any), exploit, profit/grief/theory, minimal tx shape, fix recommendation, suggested test, and **Status** with a commit/PR link once resolved.

Status values:

- **Open.** Finding stands; no code change has landed.
- **Fixed.** Code change has landed and the regression test exists. Cite commit / PR.
- **Acknowledged.** Accepted residual; not fixed because (a) blocked by off-chain assumption already documented in the threat model, or (b) cost-of-fix outweighs the realistic exploit window. The finding remains visible.
- **Won't fix.** Actively rejected as a finding (e.g. spec-as-intended; misread).

### 3.1 High-severity

#### F-1 (High, High): `fee_contract.PayMixFee` must require a `Mix` redeemer on `mix_logic`

**Status:** Fixed (commit [`46be74b`](https://github.com/logical-mechanism/Lovejoin/commit/46be74b)).

**File:** [contracts/validators/fee_contract.ak](../contracts/validators/fee_contract.ak) §`validate_pay_mix_fee`.

Negative regression tests live in [contracts/validators/fee_contract.test.ak](../contracts/validators/fee_contract.test.ak) (`pay_mix_fee_rejects_owner_redeemer_n2`, `pay_mix_fee_rejects_owner_redeemer_n6`, `pay_mix_fee_rejects_missing_mix_logic_redeemer`). Re-derived adversarially in this audit and confirmed: an Owner-branch withdraw can no longer satisfy PayMixFee, so the rotational fee-shard drain (Owner-spend two attacker mix-boxes, output two fresh attacker mix-boxes, harvest `tx.fee` from the shard) is structurally blocked.

**Original violated invariant:** "PayMixFee can only fire as part of a real Mix tx."

**Why the previous version was reachable.** The old `validate_pay_mix_fee` only checked `mix_input_count >= 2`. `mix_logic.withdraw` accepts `Owner` for any `n >= 1` well-formed mix inputs with no signer requirement. So an Owner tx with two attacker-owned mix-boxes structurally satisfied the gate, and the attacker could rotate the same value through fresh datums to drain `tx.fee` from a shard per cycle.

**Fix shape.** The validator hoists a redeemer-shape check above the input fold:

```aiken
expect Some(mix_logic_redeemer_data) =
  pairs.get_first(self.redeemers, Withdraw(Script(reference_datum.mix_logic_script_hash)))
expect mix_logic_redeemer: MixLogicRedeemer = mix_logic_redeemer_data
expect Mix { .. } = mix_logic_redeemer
```

`expect`-cast to `MixLogicRedeemer` aborts on Owner-shape data; the explicit `Mix { .. }` pin then selects the right constructor. Hoisted above the input fold so attacker txs reject before the fold.

**Suggested test.** Already present (above). Audit re-ran the negative regression tests against current `main`; they remain green.

**Hyperstructure conflict?** No.

### 3.2 Medium-severity

#### F-2 (Medium, High): `mix_box.spend` and `mix_logic` agree that only inline `MixDatum`s are well-formed

**Status:** Fixed (commit [`a77c9cb`](https://github.com/logical-mechanism/Lovejoin/commit/a77c9cb)).

**Files:** [contracts/validators/mix_box.ak](../contracts/validators/mix_box.ak), [contracts/lib/lovejoin/mixbox.ak](../contracts/lib/lovejoin/mixbox.ak).

Regression tests in [contracts/validators/mix_box.test.ak](../contracts/validators/mix_box.test.ak): `mix_box_recovery_for_no_datum`, `mix_box_recovery_for_datum_hash`, `mix_box_recovery_for_datum_hash_with_witness_data`. Re-derived in this audit and confirmed: `mix_box.spend` walks `tx.inputs` to find its own input, inspects the typed `Datum` field, and only requires `mix_logic` withdraw-zero when the datum is `InlineDatum(d)` AND `d` decodes as well-formed `MixDatum`. `DatumHash(_)` and `NoDatum` mix-script inputs take the recovery path, exactly matching `mix_logic.collect_well_formed_mix_inputs`.

**Original violated invariant:** "Every mix-box spent via the privacy pool requires a proof bound to its `(a, b)`."

**Original asymmetry.** Cardano's spend handler delivers `Some(data)` for both `InlineDatum(data)` and `DatumHash(h)`-with-witness. The previous `mix_box.spend` couldn't tell them apart from the `Option<Data>` parameter, so it gated both through "require `mix_logic` withdraw-zero". `mix_logic.collect_well_formed_mix_inputs` walked the typed `Datum` enum and silently dropped `DatumHash(_)`. Result: a `DatumHash` mix-script input was consumed without a proof.

**Fix shape.**

```aiken
expect Some(own_input) = list.find(self.inputs, fn(i) { i.output_reference == utxo })
when own_input.output.datum is {
  InlineDatum(inline_data) ->
    when try_decode_well_formed_data(inline_data) is {
      Some(_md) -> pairs.has_key(self.withdrawals, Script(mix_logic_script_hash))
      None -> True            // Inline but malformed, recovery
    }
  _ -> True                   // NoDatum / DatumHash, recovery
}
```

The `expect Some(own_input)` is a defense-in-depth abort: Cardano guarantees the spending input is on `tx.inputs`, so the `None` arm is unreachable in well-formed contexts.

**Suggested test.** Already present. Audit re-ran them; green.

**Hyperstructure conflict?** No. Recovery semantics are now consistent across spend and withdraw.

#### F-3 (Medium, High): `MixDatum` well-formedness does not reject `b == infinity`

**Status:** Won't fix on chain. SDK guard added.

**File:** [contracts/lib/lovejoin/mixbox.ak](../contracts/lib/lovejoin/mixbox.ak) §`decode_two_byte_fields`, [contracts/validators/mix_logic.ak](../contracts/validators/mix_logic.ak) §`validate_owner` (Schnorr verify).
**Confirmed by:** [contracts/lib/lovejoin/bls.test.ak](../contracts/lib/lovejoin/bls.test.ak) `point_from_bytes_accepts_infinity` (`bls12_381_g1_uncompress` accepts the canonical infinity encoding `0xc0 || 0x00 * 47`).
**SDK guard:** [offchain/src/tx/deposit.ts](../offchain/src/tx/deposit.ts) `assertNonIdentityMixPoint` runs after `aPoint`/`bPoint` are computed in `planDepositTx`, `planBulkDepositTx`, and `deriveOwner`. Test: [offchain/test/tx/deposit.test.ts](../offchain/test/tx/deposit.test.ts) `rejects a deposit's a == identity (audit F-3 SDK guard)` confirms the throw.

**Why on-chain rejection was reconsidered.** The original recommendation was to reject `a == identity` / `b == identity` in `decode_two_byte_fields`. Re-analyzing under Lovejoin's no-entry-logic design: validators only run at spend, never at deposit. So whether the box is `b == identity` or rejected by the decoder, the depositor's outcome is identical:

- **Today (no on-chain check):** anyone constructs a forged Schnorr proof on the box, sweeps via the Owner branch.
- **With on-chain check:** `try_decode_well_formed_data` returns `None`, mix_box hits Rule-2 recovery, anyone sweeps with no proof at all.

Both end with the depositor's funds gone. The "fix" only relabels the failure path (forgery sweep → Rule-2 sweep). Net security delta for the depositor: zero. So an on-chain redeploy purely to add this check buys nothing.

**Where the real defense lives.** The deposit construction is the only choke point. A legitimate user deposits with the SDK, and the SDK is now strict: `assertOwnerSecret` rejects `x ∈ {0, ≥ r}`, the rerandomization scalar `d` is rejected unless `d ∈ [1, r)`, and the new `assertNonIdentityMixPoint` is a defense-in-depth check on the COMPUTED `a` and `b` after `[d]·g` and `[x]·a`. Together these make every realistic SDK path produce `a ≠ identity` and `b ≠ identity`.

The chain remains permissive (a malicious / buggy third-party SDK could still publish such a box, and anyone could sweep it). That is the hyperstructure tradeoff: on-chain protection extends only to depositors who used a faithful client. Documented under §4.8 "Off-chain assumptions" point 1.

**Original adversarial detail (for the record).**

If `b == identity` and `a != identity`, the Schnorr equation `[z]·a == t + [c]·b` reduces to `[z]·a == t + [c]·identity == t`, so any `(z = r, t = [r]·a)` for fresh `r` verifies against any `c`. Anyone forges a valid Owner proof. Such a box appears only via `x = 0` (CSPRNG probability `1/r`, or a buggy/malicious SDK) or a hand-tampered deposit datum. The HKDF derivation in [offchain/src/wallet/seed.ts](../offchain/src/wallet/seed.ts) `deriveOwnerSecret` already rejects `x = 0`; the new SDK guard closes the remaining path (a malicious caller bypassing the secret derivation but still going through `planDepositTx`).

**Hyperstructure conflict?** No, the SDK guard sits entirely at the off-chain boundary and the on-chain validator behaviour is unchanged.

#### F-4 (Medium, High): Owner Schnorr proof not bound to the spending input's `OutputReference`

**Status:** Acknowledged (open).

**Files:** [contracts/validators/mix_logic.ak](../contracts/validators/mix_logic.ak) §`validate_owner`, [offchain/src/tx/withdraw.ts](../offchain/src/tx/withdraw.ts) §`computeOwnerCtx`.

No fix landed. The prior audit recommended adding `serialise_data(self.inputs[].output_reference)` to the Owner ctx; the v1 internal review's "no medium findings on validators" disposition implicitly defers it. Re-derived in this audit and confirmed still reachable when (and only when) two mix-boxes share `(a, b)`. The standard SDK derives `x_i` from a per-deposit HKDF index, so the precondition (duplicate `(a, b)`) requires a buggy / forked SDK or an extreme CSPRNG collision (1/r). The validator-side fix is queued for the next-redeploy bundle as Q-1 in [docs/next-redeploy.md](next-redeploy.md). Outcome is pure griefing (funds always reach the user's chosen destination; attacker pays tx fee for no monetary gain), so the bundle gates on other triggers rather than a hot-fix redeploy.

**Violated invariant:** "Each box is bound to its own proof, and a proof cannot be reused across UTxOs."

**Reachability.** Owner ctx is `blake2b_256(serialise(tx.outputs) || mix_script_hash)`. It does NOT commit to the input's `OutputReference`. If the same `(a, b)` appears on multiple unspent UTxOs at the mix script (user reused secret `x`), an attacker who observes a published Owner proof can construct a new tx that spends a different box with the same `(a, b)` to the same outputs, the proof verifies because `(a, b)` and `ctx` match.

**Concrete exploit.**

1. User `U` reuses secret `x` across two deposits → boxes `B1` and `B2` both have datum `(g, [x]g)`.
2. `U` submits an Owner withdraw of `B1` paying address `D` for `denom_lovelace`. Proof π is bound to `(a, b, ctx_outputs_to_D)`.
3. Adversary pulls π from the mempool / chain. Builds a tx spending `B2` to the same `D`, copying π.
4. `B2` is consumed; `D` receives an additional `denom_lovelace`. Adversary pays only the tx fee.

**Effect.** `U` loses `B2` (the funds DO arrive at `D`, but `U` did not authorise spending `B2`). Attacker pays tx fee for no monetary gain. Pure griefing. Could be used to (a) accelerate `U`'s anonymity loss by concentrating their funds at `D`, (b) force `U` to change destination for future withdraws, or (c) burn `U`'s mempool inclusion ordering.

**Pre-condition.** `U` must reuse `x` across deposits. The standard SDK's HKDF index `i` makes this negligible. Exposure: third-party SDKs / UI bugs / fallback CSPRNG paths.

**Recommended fix.** Bind the Owner ctx to the spending input(s):

```aiken
let input_refs_bytes =
  builtin.serialise_data(
    self.inputs |> list.map(fn (i) { i.output_reference })
  )
let ctx_preimage =
  outputs_bytes
    |> bytearray.concat(input_refs_bytes)
    |> bytearray.concat(reference_datum.mix_script_hash)
let ctx = builtin.blake2b_256(ctx_preimage)
```

Cost: one extra `serialise_data` over a small list (32 + 4 bytes per input) per `validate_owner` call. The TS prover (`computeOwnerCtx` in [offchain/src/tx/withdraw.ts](../offchain/src/tx/withdraw.ts)) must mirror it byte-for-byte; encoding-parity tests cover the wire format.

Cheaper but uglier alternative: bind to a single canonical sort of the input refs. Same byte cost, simpler list-element type, but the TS-side serialise needs to match exactly, which adds a parity vector to maintain.

**Suggested test.** Build two mix-boxes with identical `(a, b)`. Submit an Owner withdraw of box 1 with proof π. Reuse π in a second Owner withdraw of box 2 with the same outputs. Today (no fix) this passes. After fix, fails.

**Hyperstructure conflict?** No, strictly tightens binding.

#### F-5 (Medium, Medium): `mix_logic.publish` Conway certificate matrix has not been exhaustively tested

**Status:** Fixed (test-only).

**File:** [contracts/validators/mix_logic.ak](../contracts/validators/mix_logic.ak) §`publish`. The handler accepts only `RegisterCredential` and returns `False` for everything else.
**Test:** [contracts/validators/mix_logic.test.ak](../contracts/validators/mix_logic.test.ak) covers every Aiken stdlib `Certificate` variant: `publish_allows_register_credential` (positive) plus 10 negative tests (`publish_rejects_unregister_credential`, `…_delegate_credential`, `…_register_and_delegate_credential`, `…_vote_delegation`, `…_register_delegate_representative`, `…_update_delegate_representative`, `…_unregister_delegate_representative`, `…_register_stake_pool`, `…_retire_stake_pool`, `…_authorize_constitutional_committee_proxy`, `…_retire_from_constitutional_committee`).

The validator behaviour was already correct (the `_ -> False` arm rejects every non-Register variant by construction). The missing tests were belt-and-braces; adding them locks the matrix so a future Aiken stdlib bump that introduces a new Certificate variant lands as a CI failure, not a silent acceptance. Design intent: the `mix_logic` credential is for withdraw-zero only. It should never delegate, vote, register as a DRep, hold a stake pool, or sit on the constitutional committee.

**Why this matters.** In Conway, a script credential can fire validators under several `ScriptPurpose`s (Withdraw, Publish, Voting, Proposing). The validator's `else(_) { fail }` catches Voting and Proposing, fail-closed, correct. The `_ -> False` arm in `publish` catches every Certificate variant other than `RegisterCredential`, also correct, but not test-locked.

The current Aiken stdlib at [contracts/build/packages/aiken-lang-stdlib/lib/cardano/certificate.ak](../contracts/build/packages/aiken-lang-stdlib/lib/cardano/certificate.ak) defines: `RegisterCredential`, `UnregisterCredential`, `DelegateCredential`, `RegisterAndDelegateCredential`, `RegisterDelegateRepresentative`, `UpdateDelegateRepresentative`, `UnregisterDelegateRepresentative`, `RegisterStakePool`, `RetireStakePool`, `AuthorizeConstitutionalCommitteeProxy`, `RetireFromConstitutionalCommittee`. We test 2 of 11.

**Recommended fix.** Add per-variant negative tests in [contracts/validators/mix_logic.test.ak](../contracts/validators/mix_logic.test.ak) for every non-Register variant. Cheap (one test per ~5 lines).

**Hyperstructure conflict?** No.

#### F-8 (Low, Medium): `mix_logic` does not bound N from above for Owner; Mix is bounded only by `n <= 255`

**Status:** Won't fix.

**File:** [contracts/validators/mix_logic.ak:189](../contracts/validators/mix_logic.ak#L189) (`expect n <= 255` on Mix, an encoding constraint from the 1-byte sigma-OR header), [contracts/validators/mix_logic.ak:101-103](../contracts/validators/mix_logic.ak#L101) (Owner: only `n >= 1` and `length(proofs) == n`).

The per-tx CPU/mem budget enforced by the chain is the natural ceiling. A too-big tx burns through the budget mid-validator and fails. Failed-Plutus semantics consume only the collateral input, not the spending inputs. Whose collateral is on the line determines whether this is a real grief vector:

- **Mix tx:** collateral is supplied by the external provider (giveme.my). The provider's `signTxBody` endpoint ([offchain/src/tx/collateral.ts](../offchain/src/tx/collateral.ts) `GivemeMyProvider.signTxBody`) posts the full tx CBOR to their server, which evaluates the tx before co-signing. Their flag-and-validate gate refuses to sign a tx that would fail under the current ledger budget. So an oversized Mix never reaches chain. The "grief the provider" angle is structurally blocked by the provider's own self-interest plus their pre-sign validation, not by the validator.
- **Owner withdraw:** the user's own wallet supplies collateral (or the box itself under `feePayer="box"`). An oversized Owner tx burns the user's own collateral. Self-griefing only: the user has to own the boxes, hand-construct N proofs, and submit a tx they should know will fail. No third-party exposure.

There is no realistic exploit path. The original recommendation to re-add `n <= max_n` was meaningful only under the assumption that the collateral provider could be griefed; once you observe their pre-sign validation gate, the on-chain check would only protect the user from their own SDK, which is what the SDK is for. Removed from the next-redeploy bundle.

#### F-13 (Informational): Sigma-OR `c_i` length is not explicitly checked at proof boundary; `xor32` enforces 32 indirectly via the `xor_bytearray(False, ...)` builtin

**Status:** Won't fix (by design).

**File:** [contracts/lib/lovejoin/sigma_or.ak](../contracts/lib/lovejoin/sigma_or.ak) §`xor32` (lines 85-87) and the XOR fold inside `verify_pre`.

The XOR fold seeds with 32-byte `zero32` and uses `xor_bytearray(False, ...)` which aborts on length mismatch. By induction, every `acc` is 32 bytes, and a non-32-byte `br.c` aborts the fold on first iteration. The redundant `expect bytearray.length(left) == 32` was dropped in [`f2b9afa`](https://github.com/logical-mechanism/Lovejoin/commit/f2b9afa) (`perf(sigma_or): H3 drop redundant length-32 expect in xor32`); the implicit length enforcement is unchanged. ✓.

**Status:** Won't fix (by design).

## 4. Cross-cutting checks

### 4.1 Double-satisfaction analysis

- A single shard input cannot satisfy both `PayMixFee` and `Replenish` (one redeemer per input).
- A single mix-box input cannot satisfy both Owner and Mix (one withdraw redeemer per tx, applied tx-globally; the redeemer field is a sum type).
- Two fee-shards in one tx: both spend handlers run, each independently fails the `fee_input_count == 1` check. ✓.
- Two reference UTxOs holding the legit NFT: impossible (one-shot mint). The lookup additionally `expect [_]`-asserts uniqueness. ✓.
- Mix-logic withdraw-redeemer + fee-contract spend redeemer in the same tx: the F-1 fix forces the mix-logic redeemer to be `Mix` for any PayMixFee shard consumption. Owner + Replenish coexist (Replenish doesn't gate on the mix-logic redeemer at all, which is correct, Replenish is a top-up, not a fee-pay). ✓.

### 4.2 Replay & uniqueness

- Mix proofs are bound to all N output `(a',b')` + values + mix_script_hash. Replay attempts to spend the same input set produce the same tx (idempotent). Replay across different input sets fails because each input's `(a, b)` is factored into its proof verification.
- **Owner proofs are NOT bound to input UTxOs (F-4 open).** Same `(a, b)` across multiple boxes lets the proof apply to any of them. SDK-side uniqueness via HKDF index makes this unreachable in practice but the validator-level binding is loose.
- Sigma-OR proof `c_i` is XORed; per-branch `c`s are raw 32 bytes, no field-element collision concern.
- One-shot mint binds to the seed `OutputReference`; cannot fire twice.

### 4.3 Reference-input attacks

- Attacker substitutes another UTxO with the same NFT: impossible (NFT is one-shot).
- Attacker omits the reference UTxO: `read_reference_datum` aborts. ✓.
- Attacker includes an extra UTxO at the same address but with no NFT: filtered out by `holds_reference_nft`. ✓.
- Attacker includes the legit reference UTxO via `tx.inputs` instead of `tx.reference_inputs`: the always-False `reference_holder` validator rejects. The validator code only inspects `tx.reference_inputs`, so passing it as a regular input has no effect on the read. ✓.
- Attacker tampers with the reference UTxO's datum: impossible (always-False prevents the spend that would produce a tampered output).

### 4.4 Value preservation

- Mix outputs: forced ada-only at `denom_lovelace` per `walk_outputs`. ✓.
- Fee output: forced ada-only via `expect_ada_only_lovelace`. ✓.
- Native asset injection into mix/fee inputs: such UTxOs are at protocol script addresses and were created by the protocol (which forces ada-only on outputs); accidentally-parked dirty UTxOs at these addresses go through Rule-2 recovery. If somehow a unit-datum fee shard gained a native token via a bug in the protocol-side build, every PayMixFee/Replenish would abort on `expect_ada_only_lovelace(own_input.output.value)`, funds locked. The protocol cannot produce such a UTxO, so this is theory.
- Datum identity preservation: `is_unit_inline_datum(fee_output.datum)` enforces the fee output stays unit. ✓.

### 4.5 Output-position rules

- N mix outputs at positions 0..N−1 of `self.outputs`: enforced by `walk_outputs`. ✓.
- No mix outputs after position N−1: enforced by tail check `expect !output_at_script(...)`. ✓.
- Fee output position: not constrained, `fee_contract` filters and asserts `[_]`. ✓.

### 4.6 Crypto-specific

- Schnorr (Owner) ctx commits to outputs + mix_script_hash. **Does NOT commit to inputs (F-4 open).**
- Sigma-OR (Mix) ctx commits to first-N output datums + first-N output values + mix_script_hash. ✓ Note: fee output deliberately excluded (its value depends on tx.fee which depends on proof size, circular). Documented in spec.
- FS hash is `blake2b_256` over a domain-separated, length-prefixed (1-byte N for sigma-OR) concatenation. Correct.
- Statement IDs (`0x01`/`0x02`/`0x03`) provide cross-protocol domain separation. ✓.
- Subgroup checks via `bls12_381_g1_uncompress` are guaranteed by the Aiken builtin (per spec). The infinity-point IS accepted, see F-3.
- Pre-uncompression of statement points in `validate_mix` (M4 perf landing): uncompressed `G1Element`s are used for curve equations; raw bytes only feed the FS hash preimage where subgroup membership is irrelevant. No subgroup-check bypass. Verified in [docs/security-review-v1.md](security-review-v1.md) §"Validators".

### 4.7 Encoding parity (build-blocker per `docs/spec/12-build-guide.md` §Risk 1)

This audit re-confirms parity is correct:

- TS-side parity tests at [offchain/test/crypto/encoding-parity.test.ts](../offchain/test/crypto/encoding-parity.test.ts) green over 1000 vectors per N ∈ {2,3,4,6,8} for Schnorr / DHTuple / sigma-OR, per [docs/security-review-v1.md](security-review-v1.md) §"Process".
- Aiken-side parity KAT at [contracts/lib/lovejoin/encoding_parity_kat.test.ak](../contracts/lib/lovejoin/encoding_parity_kat.test.ak) green via `aiken check`.
- Value-serialise parity at [contracts/lib/lovejoin/value_serialise_parity.test.ak](../contracts/lib/lovejoin/value_serialise_parity.test.ak) confirms `assets.from_lovelace(denom)` and a real ada-only `Value` at `denom` lovelace serialise byte-identical, load-bearing for the `denom_value_bytes` cache in `validate_mix`.

A silent encoding regression breaks every proof; CI must keep these green on every release.

### 4.8 Off-chain assumptions that the chain does NOT enforce

The protocol relies on the SDK / UI for several behaviors that are NOT validated on chain. Each is a place where a malicious / buggy SDK can hurt users:

1. **`x` is uniformly random in `Z_r`.** The chain accepts any 32-byte scalar including `0` and `1`. A buggy SDK that produces `x = 0` makes `b = identity` and the box becomes free-for-all (F-3). A SDK that produces `x = constant` across users makes those boxes interchangeable (F-4 amplifier).
2. **`y_i` per Mix is fresh.** If `y_i = 1`, output equals input, proofs still pass (one branch is the identity), but no privacy is gained. Pure self-griefing.
3. **Output permutation in Mix is uniformly random over the N positions.** The chain doesn't enforce shuffling. A non-shuffling SDK leaks input-output correspondence.
4. **The depositor's `(a, b)` actually has `b = [x]·g`.** Chain doesn't check this either at deposit time, the depositor could deposit garbage `(a, b)` (anything well-formed), and only later when they try to Owner-withdraw will the math fail. Funds locked. User-error case.
5. **The `tx.fee` claimed in the tx body is honored by the ledger.** The validator trusts `self.fee` as reported by the ledger context, no way to forge it.
6. **The collateral provider doesn't deliberately fail their own non-mix scripts.** Out-of-scope here; covered in §A6 of threat model.
7. **Mix outputs share the canonical dApp stake credential.** Coordination concern only, the indexer relies on it. No on-chain enforcement (F-21). Same for the fee-script outputs (F-22).
8. **N is selected uniformly at random over the available pool.** The chain accepts any `N >= 2` for Mix (and any `N >= 1` for Owner, F-8). The SDK's job to pick `N` honestly.

## 5. Spec drift section

The implementation diverges from [docs/spec/03-contracts.md](spec/03-contracts.md) in three places. Each is reconciled here as either code-side (the spec is right, code should change) or spec-side (the code is right, spec should change).

### 5.1 `ProtocolParams.max_n` removed from on-chain datum

[contracts/lib/lovejoin/types.ak](../contracts/lib/lovejoin/types.ak) removed it during the M2 refactor. Comment in `types.ak` lines 23-27 explains why: hard-coding a max would freeze the protocol if Cardano's per-tx limits ever rise. The natural ceiling is the chain CPU budget; SDKs can carry a recommended N as off-chain config (`network.<net>.json`).

**Resolution: spec-side.** Update [docs/spec/03-contracts.md](spec/03-contracts.md) §1 ProtocolParams to drop `max_n`, with a note that `mix_logic` only enforces `n >= 2` for Mix (and `n <= 255` from the 1-byte sigma-OR header) and `n >= 1` for Owner. Already partially noted in [docs/spec/01-protocol.md](spec/01-protocol.md); finish the sweep across §03.

This audit does NOT do the spec edit, that's a follow-up for a docs-touching PR. The drift is documented here so the next reader knows.

### 5.2 `ProtocolParams.fee_shard_target` removed from on-chain datum

Same situation. [docs/spec/03-contracts.md](spec/03-contracts.md) §1 lists `fee_shard_target: Int`; the implementation dropped it. Comment in [contracts/lib/lovejoin/types.ak](../contracts/lib/lovejoin/types.ak) lines 29-32: no validator reads it, committing it on-chain forced every reference-datum decode to walk one more field, dropped during the M4.5 schema cleanup.

**Resolution: spec-side.** Same as 5.1. Update the spec to drop the field, with a note that the canonical 10-shard pool is off-chain coordination.

### 5.3 Owner branch supports `n >= 1` (bulk withdraw), not exactly 1

Already reconciled in [`86fb05c`](https://github.com/logical-mechanism/Lovejoin/commit/86fb05c) (`docs(spec): align Owner branch with the implementation (N>=1)`). Verified in this audit: §2 Owner-branch text in [docs/spec/03-contracts.md](spec/03-contracts.md) now says `N ≥ 1` and cross-references the F-1 fix.

**Resolution: closed.** No drift remaining.
